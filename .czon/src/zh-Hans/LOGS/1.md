# 解放的曙光即将来临

现在是 2026-01-05。

最近开始用 [vibe-kanban](https://www.vibekanban.com/) 来做一些工程级别的项目管理尝试，发现效果还不错。

它提供了一个看板，让我能站在 Team Leader 的角度来管理任务和进度。以前在使用 Claude Code 的时候，总感觉自己是个 Senior Developer，在带一个 Junior Developer，我需要高度集中精力投入到一个任务中。Claude Code 没有 work-tree 隔离的概念，因此我没有办法同时管理多个任务。
而使用 vibe-kanban 之后，我可以把任务拆分成多个卡片，然后并行地推进它们。每个卡片都有自己的上下文和进度，我可以随时切换关注点，而不需要担心丢失信息。

这种方式让我感觉自己更像是一个“包工头”，而不是一个“码农”。我可以更好地协调各个任务，分配资源，监控进度，而不是陷入到具体的编码细节中去。

我很快发现，我的瓶颈来到了创建新任务和 Review 任务上（是的，显然还需要 Review 任务）。不安感并没有因为我用了 vibe-kanban 而消失。甚至，我觉得它变得更强烈了，因为我现在有更多的任务需要管理，而不是专注于一个任务。

简而言之，爽了一秒，然后又不爽了。

但显然 vibe-kanban 是有意义的，是好用的。但它使得我意识到一个问题：我对细节的控制欲和我对快速推进的渴望之间存在矛盾。而这个矛盾本身是由我的带宽限制所导致的。以前就存在，现在只是变得更强烈了。

于是我开始思考，为什么我有这么强的控制欲呢？如果我不能放下对细节的控制欲，我就无法真正地 scaling up。我终归是一个有限的人类，我的带宽是有限的。思考了一下技术主管们，他们并不会亲自写代码，而是通过管理和协调来推动项目进展。他们怎么敢放下对细节的控制欲呢？只会招人显然不够，他们需要一套系统来确保项目的质量和进度。人和 AI 作为底层劳动力，其实本质上可靠性并无太大区别。人类也会犯错，AI 也会犯错。关键在于如何设计一个系统，使得整体的可靠性和效率最大化。

你可以参考 [这篇文章](../INSIGHTS/2.md) ，我在里面详细讨论了这个问题，并提出了一些解决方案。

另外，我还意识到 vibe-kanban 的生态位不同于 Claude Code。它是高于 Claude Code / CodeX 以及同类产品的。也就是说，Claude Code 们的进步，带给 vibe-kanban 的也是进步，而不是威胁。因为它们处于不同的抽象层级。Claude Code 们是底层劳动力，而 vibe-kanban 是管理工具。这种分工使得它们可以协同工作，而不是相互竞争。

但是，vibe-kanban 仍然是 team-leader 级别的工具，而不是 CEO 级别的工具。CEO 需要关注的是战略和方向，而不是具体的任务和进度。因此，我还需要找到一种方法，来提升我的战略思维能力，而不是仅仅依赖于任务管理工具。或许，已经有一个产品叫做 vibe-okr 了？😆

我与小伙伴讨论他近期做的 Agent 项目，我认为他之前搞的方便人类 review AI 的 Plan 的工具很有意思，但终归于解放之路还是南辕北辙。他的工具是最终还是增强了控制欲，而不是放下控制欲。我认为这是一种误区。真正的解放，是要学会放下对细节的控制欲，而不是试图通过工具来增强控制欲。他们的工具可能会让人觉得更有掌控感，但实际上并没有真正地解放人类的带宽。

公司级别(?) > 团队级别(vibe-kanban) > 个人级别(claude code, ...etc).

总之，还有很多工作要做，但我觉得我已经找到了一条可行的道路。
