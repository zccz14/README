现在是 2026 年 1 月 28 日，周三，晚上。

昨天将 CZON 推销给 GB ， GB 表示有兴趣，然而他遇到了 call OpenAI API 需要 HTTP Proxy 的问题，但是之前 CZON 不支持 HTTP Proxy，所以我今天花了一些时间给 CZON 加上了 HTTP Proxy 的支持。等待 GB 的进一步反馈。

 GB 是继 C1 之后的第二个 CZON 用户。我准备测试 CZON 的落地情况。这个时候不能操之过急，必须稳扎稳打，先把早期用户的体验做好，等到有了一定的用户基础之后，再考虑大规模推广。

现在 CZON 的闭环还是略微有些生硬。我考虑完全使用 GitHub Actions 来实现 CZON 的闭环。也就是说，用户提交内容源文件到主分支之后，GitHub Actions 会自动触发 CZON 的运行，然后将结果直接推回到主分支内（或者提 PR）。不过这样需要配置 GitHub Secrets 来存储 OpenAI API Key 等。这样可以省去用户在本地运行 CZON 的麻烦，降低使用门槛。（当然，这实际上还是没有到 CZONE 那种完全无门槛的方式，因此我也在纠结是否要做这些优化）

CZONE 被定位是一个完全无门槛的产品。用户只需要在 GitHub 上注册账户，然后通过 GitHub OAuth 登录 CZONE，就可以直接创建内容站，写作，发布静态站点，完全不需要跳转到其他地方。移动端编辑体验与发朋友圈或者发推特类似，非常简单直接。而桌面端的编辑体验则类似于使用 Notion 或者 Typora 之类的所见即所得的编辑器。用户只需要专注于内容创作，其他的事情全部交给 CZONE 来处理。CZONE 的好处在于，底层基于 Markdown 和 Git，用户的数据完全属于用户自己，用户可以随时导出数据，或者将数据迁移到其他平台上去。这是符合 Web3 精神的。

如果今天 GB 使用的是 CZONE 而不是 CZON，那么他就完全不需要考虑 HTTP Proxy 的问题。

我想我应该还可以新增几个用户来测试 CZON 的落地情况。
