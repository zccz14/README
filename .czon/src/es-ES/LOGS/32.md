Hoy es 28 de enero de 2026, miércoles, por la noche.

Ayer le presenté CZON a GB, quien mostró interés. Sin embargo, se encontró con el problema de que para llamar a la API de OpenAI necesita un HTTP Proxy, pero CZON no lo soportaba previamente. Así que hoy dediqué algún tiempo a añadir soporte para HTTP Proxy en CZON. Estoy esperando más comentarios de GB.

GB es el segundo usuario de CZON después de C1. Estoy preparándome para probar la implementación de CZON. En este momento no hay que precipitarse; debemos ser firmes y constantes, asegurándonos primero de que la experiencia de los primeros usuarios sea buena. Una vez que tengamos una base de usuarios sólida, podremos considerar una promoción a gran escala.

El flujo de trabajo cerrado de CZON sigue siendo un poco rígido. Estoy considerando implementar completamente el ciclo de CZON usando GitHub Actions. Es decir, después de que un usuario envíe los archivos fuente del contenido a la rama principal, GitHub Actions activaría automáticamente la ejecución de CZON y luego enviaría los resultados directamente de vuelta a la rama principal (o crearía una Pull Request). Sin embargo, esto requeriría configurar GitHub Secrets para almacenar la clave API de OpenAI, etc. Esto eliminaría la molestia para los usuarios de ejecutar CZON localmente, reduciendo la barrera de entrada. (Por supuesto, esto aún no alcanza el nivel de CZONE, que es completamente sin barreras, por lo que también estoy debatiendo si hacer estas optimizaciones).

CZONE está posicionado como un producto completamente sin barreras. Los usuarios solo necesitan registrarse en GitHub, luego iniciar sesión en CZONE mediante OAuth de GitHub, y podrán crear directamente un sitio de contenido, escribir y publicar un sitio estático, sin necesidad de saltar a otros lugares. La experiencia de edición en dispositivos móviles es similar a publicar en un "momento" (como en WeChat) o en Twitter, muy simple y directa. La experiencia de edición en escritorio es similar a usar editores WYSIWYG como Notion o Typora. Los usuarios solo necesitan concentrarse en crear contenido; CZONE se encarga de todo lo demás. La ventaja de CZONE es que, al estar basado en Markdown y Git, los datos del usuario le pertenecen completamente. Los usuarios pueden exportar sus datos en cualquier momento o migrarlos a otras plataformas. Esto está en línea con el espíritu de Web3.

Si hoy GB estuviera usando CZONE en lugar de CZON, no tendría que preocuparse en absoluto por el problema del HTTP Proxy.

Creo que aún podría agregar algunos usuarios más para probar la implementación de CZON.
