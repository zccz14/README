Ayer le presenté CZON a GB, quien mostró interés. Sin embargo, se encontró con el problema de que para llamar a la API de OpenAI necesita un proxy HTTP, pero CZON no lo soportaba. Así que hoy dediqué algún tiempo a añadir soporte para proxy HTTP en CZON. Estoy esperando más comentarios de GB.

GB es el segundo usuario de CZON después de C1. Estoy preparándome para probar la implementación de CZON. En este momento no hay que precipitarse, debemos avanzar con paso firme, primero asegurar una buena experiencia para los primeros usuarios, y luego, una vez que tengamos una base de usuarios sólida, considerar una promoción a gran escala.

El ciclo de CZON todavía es un poco rígido. Estoy considerando implementar completamente el ciclo de CZON usando GitHub Actions. Es decir, después de que un usuario envíe los archivos fuente del contenido a la rama principal, GitHub Actions activará automáticamente la ejecución de CZON y luego enviará los resultados directamente de vuelta a la rama principal (o creará un PR). Sin embargo, esto requiere configurar GitHub Secrets para almacenar la clave API de OpenAI, etc. Esto puede ahorrar a los usuarios la molestia de ejecutar CZON localmente, reduciendo la barrera de entrada. (Por supuesto, esto aún no llega al nivel de CZONE, que es completamente sin barreras, por lo que también estoy dudando si hacer estas optimizaciones)

CZONE está posicionado como un producto completamente sin barreras. Los usuarios solo necesitan registrarse en GitHub, luego iniciar sesión en CZONE a través de OAuth de GitHub, y pueden crear directamente un sitio de contenido, escribir y publicar un sitio estático, sin necesidad de saltar a otros lugares. La experiencia de edición en dispositivos móviles es similar a publicar en redes sociales o en Twitter, muy simple y directa. Y la experiencia de edición en escritorio es similar a usar editores WYSIWYG como Notion o Typora. Los usuarios solo necesitan concentrarse en crear contenido, todo lo demás lo maneja CZONE. La ventaja de CZONE es que, al estar basado en Markdown y Git, los datos del usuario le pertenecen completamente, pueden exportarlos en cualquier momento o migrarlos a otras plataformas. Esto está en línea con el espíritu de Web3.

Si hoy GB estuviera usando CZONE en lugar de CZON, no tendría que preocuparse en absoluto por el problema del proxy HTTP.

Creo que aún podría agregar algunos usuarios más para probar la implementación de CZON.
