現在は2026年1月19日、月曜日の午後です。

今日はとても遅く起きました。昨夜、CZONEとOpenCodeをいじくり回していて興奮していたからです。結果は満足のいくものではありませんでしたが、もしあの手を動かしていなければ、昨日の結果はもっと失敗していたかもしれません。

> 夜更かしは、その一日の失敗を認めたくないからだ。
> —— from PH

昨夜、OpenCode + MiniMax M2.1を使って、CZONE（CZONのオンライン版）をゼロから構築してみました。詳細は[こちらのログ](czon://ec05a61746e4d7898feb28f18729f0ab3df43b67569d88f217ae6472505c5f66)をご覧ください。

AIは次々と質問を投げかけ、技術選定からスキャフォールディング構築、機能設計、そしてCI/CDフローまで、プロセス全体が非常に速く進みました。

正直なところ、少し速すぎて、目が回りそうでした（笑）。

しかし、重要な「しかし」です。すぐに問題にぶつかりました。

GitHub REST APIの権限問題を解決する際に、AIは詳細な理解が不十分であることに気づきました。

**博識強記？そうではありません。**

例えば、リポジトリを初期化した後、`.github/workflows/pages.yml`ファイルを修正してCZONのビルドステップを追加する必要があります。これは`workflow`スコープの権限を必要とする作業ですが、OpenCodeが提供したコードにはこの権限が含まれていませんでした。これはGitHub APIのドキュメントを調べればすぐにわかることです。しかし、AIは繰り返しこの詳細を見落としました。また、GitHubも問題で、エラーメッセージは単なる404で、権限不足のヒントは一切ありませんでした。AIもこの問題に全く気づきませんでした。

この過程で、`index.md`への書き込みは成功し、`.github/index.md`への書き込みも成功し、`github/workflows/pages.yml`への書き込みも成功しましたが、`.github/workflows/pages.yml`への書き込みだけが失敗しました。会話は何度もやり取りされましたが、AIは毎回コードを修正する必要がありました。それにもかかわらず、これほど明白な状況で、`.github/workflows/`ディレクトリが特殊な権限を必要とする可能性に気づかなかったのです。これは、AIの注意力が散漫で、デバッグモードでの推論能力が十分でないことを示しています。

私は強く提案します。LLM自身、または外部制御フレームワークのエージェントには、**ラボモード**(Lab Mode)が必要です。このモードでは、エージェントは対照実験を繰り返し設計し、実験結果を検証して真実を見つけ出す必要があります。私は時々、LLMは無意識の脳のようなものだと思います。どこを指しても、そこだけが光ります。プロンプトに何が書かれているかによって、注目するものが決まります。

時には博識強記であることを望み、時には澄み切った無知であることを望みます。ある意味で、LLMが消費するエネルギーは一定であり、私たちは異なるタスクに対応する際に、エネルギーを最も必要な場所に配分してほしいのです。平均的に配分するのではなく。最近のLLM分野の進展も、しばしばこの考え方を採用しています。

**水槽の中の脳、行動力は限定的**

もう一つ、非常に重要で煩わしい理由は、OpenCodeの自己デバッグ能力が不足していることです。ブラウザを開いて操作する能力が全くないため、AIは頻繁に推測し、ログを出力し、そして私にログを見てくれるよう頼むしかありません。私は時々それに付き合って遊んでみますが、時にはAIを見て、まるで私のメンティーのようで、何を考えているのか全くわからず、ただやきもきします。私は少し不器用なメンティーを持つことは受け入れられますが、「両手のない」水槽の中の脳であることはおそらく受け入れられないでしょう。やはり、思考の閉ループを完成させる方法を考えなければなりません。隣のGoogle社のAntigravityはこの点でうまくやっているようです。Chromeと同系列だからかもしれません。

コミュニティのソリューションとしては、CypressやPlaywrightなどのエンドツーエンドテストフレームワークを使用してブラウザを操作することが、良い選択肢となるでしょう。結局のところ、現在多くの操作にはブラウザ側のインタラクションが必要であり、APIだけでは不十分なのです。

**進みすぎ、地に足がついていない**

最後に、私が原因と考える点です。今回、AIはゼロから始めて、10分足らずで何十ものファイルを書き上げました。私はAIを見て、まるで印刷機のように、全く休む間もなく感じました。しかし、どんな複雑なシステムでも、アーキテクチャが必要で、階層化が必要で、各モジュールの基礎的な品質を保証する必要があります。下層を完成させ、テストをしっかり行って初めて、上層を安心して構築できます。AIには今、このようなリズム感が全くありません。純粋にコードを印刷しているだけです。もし自己デバッグ能力が備わっていれば、柔軟に上下を修正できるかもしれませんが、本当に問題を起こさないためには、本質的には正しい概念、正しい抽象化、正しい実装に依存します。筋が通っていること、理にかなっていることに頼るのです。AIがこのことにどれだけ時間を費やすかについては、現時点ではまだ遠く及ばないと思います。おそらく、これは調整層だけが解決できることであり、LLM単体ではできません。**LLMはただ道を切り開くだけです。洪水のように、位置エネルギーの最も低いところへと流れ出していきます。**

しかし、多くの人間も同じです。昔からの古典的なジョークがあります。配管工が水漏れを修理し、ここを塞げばあちらが破裂する。頭痛には頭を、足の痛みには足を治療する。結局根本的に解決できず、水の中でボートを漕ぐしかなくなるというものです。
