現在は2026年1月28日、水曜日の夜です。

昨日、CZONをGBに紹介したところ、GBは興味を示してくれました。しかし、彼はOpenAI APIを呼び出す際にHTTP Proxyが必要な問題に直面しました。これまでのCZONはHTTP Proxyをサポートしていなかったため、本日は時間をかけてCZONにHTTP Proxyサポート機能を追加しました。GBからのさらなるフィードバックを待っています。

GBは、C1に続く2人目のCZONユーザーです。私はCZONの実用性をテストする準備を進めています。この段階では焦らず、着実に進めることが重要です。まずは初期ユーザーの体験を向上させ、一定のユーザーベースが構築されてから、大規模な普及を検討すべきです。

現在のCZONのクローズドループは、まだやや硬直しています。私は、GitHub Actionsを完全に活用してCZONのクローズドループを実現することを検討しています。つまり、ユーザーがメインブランチにコンテンツソースファイルを提出すると、GitHub Actionsが自動的にCZONを実行し、結果を直接メインブランチにプッシュバックする（またはPRを提出する）方式です。ただし、この方法ではOpenAI API Keyなどを保存するためにGitHub Secretsの設定が必要となります。これにより、ユーザーがローカルでCZONを実行する手間が省け、使用のハードルが下がります。（もちろん、これはCZONEのような完全無料の方式にはまだ及ばないため、この最適化を行うべきかどうか悩んでいます）

CZONEは、完全無料の製品として位置付けられています。ユーザーはGitHubにアカウントを登録し、GitHub OAuthでCZONEにログインするだけで、直接コンテンツサイトを作成し、執筆し、静的サイトを公開できます。他のプラットフォームに移動する必要は一切ありません。モバイル端末での編集体験は、SNSの投稿と同様にシンプルで直感的です。デスクトップ端末での編集体験は、NotionやTyporaのようなWYSIWYGエディタに似ています。ユーザーはコンテンツ制作に集中でき、その他の作業はすべてCZONEが処理します。CZONEの利点は、基盤がMarkdownとGitであるため、ユーザーのデータは完全にユーザー自身のものとなり、いつでもデータをエクスポートしたり、他のプラットフォームに移行したりできる点です。これはWeb3の精神に合致しています。

もし今日GBがCZONではなくCZONEを使用していたなら、彼はHTTP Proxyの問題を一切考慮する必要はなかったでしょう。

私は、CZONの実用性をテストするために、さらに数名のユーザーを追加すべきだと考えています。
